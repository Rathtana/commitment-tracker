<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Vercel HTTPS | All traffic over TLS; @supabase/ssr sets Secure cookie flag on HTTPS |
| Vercel → Supabase | Server-side requests from Vercel serverless functions to Supabase Postgres via DATABASE_URL (server-only env var) |
| client → Supabase Auth | Authentication via @supabase/ssr; session cookies set HttpOnly so JS cannot read them |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | Spoofing | Session cookie hijacking | mitigate | @supabase/ssr sets HttpOnly (blocks JS access) + Secure (HTTPS only) + SameSite=Lax (blocks cross-site POST) automatically on production HTTPS. Verified via DevTools in Step 4 of UAT. |
| T-04-04-02 | Elevation of Privilege | Session fixation on deploy | mitigate | middleware.ts calls supabase.auth.getUser() on every request which rotates tokens; stale session tokens are invalidated by Supabase Auth. |
| T-04-04-03 | Denial of Service | Password reset rate limit bypass | mitigate | Supabase Auth built-in rate limiting enabled by default. Confirmed active in Supabase Auth Settings (Step 6 of UAT). No app-level code needed per D-09. |
| T-04-04-04 | Information Disclosure | DATABASE_URL in Vercel env | mitigate | DATABASE_URL set WITHOUT NEXT_PUBLIC_ prefix — server-side only, not bundled into client JS. Verified at deploy time by confirming env var has no NEXT_PUBLIC_ prefix in Vercel dashboard. |
| T-04-04-05 | Tampering | Env var NEXT_PUBLIC_SUPABASE_ANON_KEY exposure | accept | Anon key is designed to be public (safe to embed in client JS); Supabase RLS enforces authorization at DB layer regardless of anon key exposure. |
</threat_model>

# Phase 2: Goals & Dashboard (Three Types) - Context

**Gathered:** 2026-04-18
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver the full daily-use loop: create, edit, and delete goals for the current or a future month across all three goal shapes (count, checklist, habit); render every current-month goal on a single scrollable dashboard with an identical pace-aware progress bar per card; and log progress in one click per type directly from the dashboard — with optimistic updates, a 6-second undo toast, PROG-04 missed-day backfill, and the POLSH-03 habit month-grid.

**In scope:**
- Drizzle schema extension: `tasks`, `habit_check_ins`, `progress_entries` child tables with RLS policies; polymorphic-validity CHECK on `goals`; migrations applied via Supabase CLI
- Pure `src/lib/progress.ts` function (shared client + server; single source of truth for % and pace math)
- Dashboard route (replaces landing stub) rendering current-month goals
- Goal create dialog (2-step: type picker → per-type fields) + edit (reopens dialog prefilled) + delete (AlertDialog confirm)
- Count `+1`/stepper, checklist task toggle, habit grid tap-today
- Undo toast via shadcn `<Sonner>` (uniform across types, 6-second window)
- Habit month-grid (calendar layout, 7×6, Sun–Sat) with historical tap-to-backfill within current month
- "Log for earlier day" affordance on count cards (date picker + stepper, constrained to current month)
- Empty dashboard state (headline + 3 greyed example cards + CTA)
- Pace-aware visualization: expected-line tick + amber "behind/ahead by N" chip, with early-month guard before day 5

**Not in scope (deferred to Phase 3 / Phase 4):**
- Month navigation (prev/next controls, URL `/dashboard/[month]` dynamic routing) — Phase 3
- Past-month read-only enforcement at the RLS / service / UI layers — Phase 3
- Future-month pre-planning UI and "Copy from last month" — Phase 3
- End-of-month reflection field (POLSH-04) — Phase 3
- Mobile-responsive final pass at 375px and error-toast hardening — Phase 4

<decisions>
## Implementation Decisions

### Schema & Migrations

- **D-01:** `tasks` table shape matches `ARCHITECTURE.md` recommendation: `id uuid PK DEFAULT gen_random_uuid()`, `goal_id uuid NOT NULL REFERENCES goals(id) ON DELETE CASCADE`, `label text NOT NULL`, `is_done boolean NOT NULL DEFAULT false`, `position int NOT NULL DEFAULT 0`, `done_at timestamptz NULL`, plus `created_at`/`updated_at`.
- **D-02:** `habit_check_ins` table shape: `goal_id uuid NOT NULL REFERENCES goals(id) ON DELETE CASCADE`, `check_in_date DATE NOT NULL`, `created_at timestamptz NOT NULL DEFAULT now()`, **composite primary key `(goal_id, check_in_date)`** — uniqueness per day is enforced structurally, not by app code.
- **D-03:** Count-goal targets live on the `goals` table itself as nullable columns: `target_count INT`, `current_count INT`. A polymorphic-validity CHECK enforces the right columns are populated per type (see D-04). This is `ARCHITECTURE.md` Pattern 1 (class-table-inheritance with shared parent).
- **D-04:** Polymorphic validity CHECK on `goals`:
  ```sql
  CHECK (
    (type = 'count'     AND target_count IS NOT NULL AND current_count IS NOT NULL AND target_days IS NULL)
    OR (type = 'checklist' AND target_count IS NULL     AND current_count IS NULL     AND target_days IS NULL)
    OR (type = 'habit'     AND target_count IS NULL     AND current_count IS NULL     AND target_days IS NOT NULL)
  )
  ```
  Plus non-negativity / positivity CHECKs: `current_count >= 0`, `target_count > 0`, `target_days > 0`.
- **D-05:** A new `progress_entries` audit/log table for count-type only: `id uuid PK`, `goal_id uuid FK ON DELETE CASCADE`, `delta int NOT NULL` (signed — supports undo + corrections), `logged_at timestamptz NOT NULL DEFAULT now()`, `logged_local_date DATE NOT NULL` (the user's local day the entry counts toward). Enables PROG-04 backfill (`logged_local_date` ≠ today) and PROG-05 undo (delete the latest row).
- **D-06:** `goals.current_count` is a denormalized cache. Every `+1` (or stepper delta) is a single transaction: `INSERT INTO progress_entries (...)` AND `UPDATE goals SET current_count = current_count + delta`. Undo reverses both in the same transaction.
- **D-07:** `tasks` and `habit_check_ins` serve as their own logs — no separate audit table. Task toggle flips `is_done` + sets/clears `done_at`. Habit mark-today inserts/deletes a row in `habit_check_ins`.
- **D-08:** RLS on the three new tables mirrors the Phase 1 pattern (D-20, D-22): Drizzle `crudPolicy()` helpers co-located with schema definitions. Ownership resolved via JOIN to `goals.user_id = auth.uid()` (no direct `user_id` on children — identity flows through `goal_id`). All four CRUD ops gated.
- **D-09:** Migration is generated by `drizzle-kit generate` into `supabase/migrations/` (Phase 1 D-09 pattern) and applied via `supabase db push --linked`. A post-push verification script (same inline `node -e` + postgres.js pattern used in Plan 01-03) asserts tables + policies + CHECKs exist.

### Progress Calculation

- **D-10:** A new pure function `src/lib/progress.ts` is the single source of truth for % and pace. Signature:
  ```typescript
  // Core shape (discriminated union input):
  computeProgress(goal: Goal, now: Date, userTz: string): {
    percent: number;          // 0..1
    raw: { done: number; total: number };
    expected: number;         // 0..1, expected at this point in month
    pace: 'on-pace' | 'behind' | 'ahead' | 'warming-up';
    paceDelta: number;        // signed difference, e.g. -2 = "behind by 2"
  }
  ```
  Imported by server code (authoritative dashboard render) and client code (useOptimistic). Pure — no DB, no timers — testable via Vitest fixtures matching Phase 1's D-23 suite.
- **D-11:** Per-type `percent` formula:
  - `count` → `current_count / target_count`
  - `checklist` → `COUNT(tasks WHERE is_done) / COUNT(tasks)`
  - `habit` → `COUNT(DISTINCT check_in_date) / target_days`
- **D-12:** Pace math: `expected = days_elapsed / days_in_month` where both are computed in the user's local timezone via `today(now, userTz)` and `monthBucket(now, userTz)` (Phase 1 D-13, D-14). `paceDelta = round((percent - expected) × target)` — a concrete integer count difference, not a percentage. Applies to `count` and `habit` only; `checklist.pace = 'on-pace'` always (no time axis).
- **D-13:** Early-month guard: when `days_elapsed < 5` (user's local month), return `pace: 'warming-up'`. The UI suppresses the pace chip and dims the expected-line tick. Prevents PITFALLS §4 demotivation.
- **D-14:** Pace chip text: `'on pace'` when `|paceDelta| <= 1`; `'behind by N'` when `paceDelta < -1`; `'ahead by N'` when `paceDelta > 1`. Never red; muted amber for behind, muted emerald for ahead.

### Goal Creation, Edit & Delete

- **D-15:** Goal creation UI = shadcn `<Dialog>` overlaying the dashboard. Motion `<AnimatePresence>` handles open/close. Single component, conditional step-2 rendering by chosen type.
- **D-16:** Two-step create flow:
  - **Step 1 (type picker):** three large example cards — "Count — Read 5 books", "Checklist — Home renovation", "Habit — Meditate daily". Clicking one advances.
  - **Step 2 (type-specific fields):** title (required, all types); then per-type: count → `target_count` (int > 0); checklist → ordered task list with inline add (min 1 task, position auto-assigned); habit → `target_days` (1–31, defaults to `days_in_month` of the goal's month).
- **D-17:** All goals created in Phase 2 land in the **current month** (computed via `monthBucket(new Date(), userTz)`). Phase 2 does NOT expose month-forward navigation — that's Phase 3. The goal dialog has no month selector.
- **D-18:** Edit UX = kebab `<DropdownMenu>` on each card → "Edit" reopens the same Dialog, skips Step 1, prefills Step 2. `type` is read-only after creation (changing type would invalidate historical progress). Optional notes, title, target_count, task list (add/remove/toggle), and target_days are editable per type.
- **D-19:** Delete UX = kebab "Delete" → shadcn `<AlertDialog>` confirm. Copy: "Delete [goal title]? This removes the goal and all its progress and can't be undone." `ON DELETE CASCADE` on `goals` handles children.
- **D-20:** Zod canonical schemas live in new `src/lib/schemas/goals.ts` as a discriminated union on `type`. Imported by client form (via `zodResolver`) and server actions (for re-validation). Mirrors `src/lib/schemas/auth.ts` pattern.

### Dashboard Layout & Card Anatomy

- **D-21:** Dashboard layout = single column, max-width container (~720px), centered on wider viewports. Scrolls vertically. Matches DASH-01 "single scrollable list, no tabs, no drill-down." Mobile and desktop read identically.
- **D-22:** Every card type shares an **identical progress-bar component**. Full-width rounded bar (h-2 or h-3), emerald-600 fill over muted track, expected-line tick overlay + muted amber pace chip. Per-type content (count stepper, task list, habit grid) sits above or below the bar. The bar is the consistent "draw" across all three types.
- **D-23:** Progress bar is a custom Motion component — `<motion.div>` with spring-physics `width` transition. **NOT** the default Radix Progress indicator. Addresses STATE.md research flag (Motion + shadcn Progress CLS regression); planner must include the pattern research pass called for there.
- **D-24:** Card anatomy (top to bottom, uniform across types):
  1. **Header:** goal title (medium weight) + kebab `<DropdownMenu>` right-aligned
  2. **Progress bar row:** bar + expected-line tick + pace chip (suppressed before day 5)
  3. **Type-specific surface:**
     - Count: primary `+1` button + inline `-/+ N` stepper + "X of Y" raw label
     - Checklist: scrollable task list with shadcn `<Checkbox>` rows + "X of Y done" label
     - Habit: month-grid + "X of N days this month" label
- **D-25:** Pace indicator visual = expected-line tick overlay at `expected × 100%` of the bar width + a small pill chip beside the bar. **Bar fill color never shifts based on pace** — stays emerald regardless. PITFALLS §1 rationale: no red, no shame.
- **D-26:** Card ordering on the dashboard = `goals.position ASC, created_at ASC`. Manual reorder UI is deferred (Claude's Discretion); the column already exists from Phase 1.

### Empty State & Onboarding

- **D-27:** Empty dashboard (zero goals in current month) = headline "It's [Month Year]. What do you want to commit to?" + three greyed example cards ("Count — Read 5 books", "Checklist — Home renovation", "Habit — Meditate daily") rendered as illustrative, **not clickable** to auto-add, + a prominent "Add your first goal" primary button that opens the standard create dialog. Matches `FEATURES.md §Onboarding` verbatim.

### One-Click Logging & Optimistic UI

- **D-28:** Count logging = primary `+1` button on card face; secondary inline `-`/`+` stepper for correction (e.g., "logged 3 at once", "tapped by mistake"). Both use React 19 `useOptimistic` — the bar moves instantly. Server action writes a `progress_entries` row and updates the `goals.current_count` cache in one transaction. On failure, `useOptimistic` auto-rolls back.
- **D-29:** Checklist logging = tap a task's `<Checkbox>` to toggle `is_done`. `useOptimistic` flips the checkbox + recomputes the bar instantly. Server action persists the flip and sets/clears `done_at`. Tasks table is its own log — no separate audit row.
- **D-30:** Habit logging = tap today's cell in the habit grid to mark/unmark. Server action inserts (on mark) or deletes (on unmark) a `habit_check_ins` row for `today(now, userTz)`. `useOptimistic` fills/empties the cell instantly.
- **D-31:** Optimistic conflict handling — if a server action returns an error, `useOptimistic` auto-rolls back. An inline toast surfaces the error (Phase 4 hardens this; Phase 2 ships the baseline behavior).

### Undo (PROG-05)

- **D-32:** Undo UX = shadcn `<Sonner>` toast after every successful progress mutation: `"Logged +1 on [goal title]. Undo"`. 6-second window. Uniform across all three types.
- **D-33:** Undo semantics by type:
  - Count → delete the latest `progress_entries` row AND decrement `goals.current_count` by its `delta`, in one transaction
  - Checklist → flip `is_done` back + clear/reset `done_at` to prior value
  - Habit → delete the `habit_check_ins` row for that date (or re-insert if undo of an unmark)
- **D-34:** Only the **most recent** mutation is undoable per goal at any time. A second mutation replaces the toast and locks in the first one. Keeps the mental model simple and avoids stacked-undo ambiguity.

### Historical Backfill (PROG-04)

- **D-35:** Habit backfill = tap any past cell within the current month on the habit grid to toggle. Future cells are dimmed and non-interactive. Sets `habit_check_ins.check_in_date` to the clicked date (not today).
- **D-36:** Count backfill = kebab menu on count cards has a "Log for earlier day" item → opens a shadcn `<Popover>` with a date picker (constrained to the current month, past days only, today excluded since the primary `+1` handles today) + a stepper. Commits a `progress_entries` row with `logged_local_date` = chosen date, `logged_at` = now, and updates the cache.
- **D-37:** Checklist backfill is **N/A** — a task check is stateful, not date-stamped; `done_at` captures when the check happened, which is fine. No PROG-04 affordance on checklist cards.

### Habit Month-Grid (POLSH-03)

- **D-38:** Grid layout = calendar-style, 7 columns (Sun–Sat) × up to 6 rows, covering the entire current month. Dead cells outside the month render as empty spacer (not clickable). Weekday header row ("S M T W T F S") sits above the grid.
- **D-39:** Cell states:
  - **Hit:** filled emerald-600
  - **Miss:** empty (muted track color)
  - **Today:** ring outline in primary color (regardless of hit/miss state)
  - **Future:** dimmed opacity, cursor: not-allowed, not tappable
- **D-40:** Accessibility — each cell is a `<button>` with `aria-label="April 3 — not done"` / `"April 3 — done"` / `"April 3 — today, not yet done"`. Keyboard-navigable.

### Claude's Discretion

- Exact Motion spring `stiffness`/`damping`/`bounce` values for the progress bar width animation (must not cause CLS regression — see D-23)
- Card hover / focus / active styles beyond shadcn defaults
- Whether task drag-reorder is supported during goal creation (minimum requirement: type-to-add + delete; drag is nice-to-have)
- Exact amber and success color token values for the pace chip (must use Tailwind v4 `@theme` tokens in `globals.css`; must NOT override core `--spacing-*` tokens per STATE.md lesson)
- Copy voicing of empty-state headline, "Log for earlier day" popover, and toast messages
- Kebab menu icon choice (lucide-react `MoreHorizontal` vs `MoreVertical`) and alignment
- Optimistic UI error-toast copy/placement (Phase 4 hardens this)
- Grid cell hover/tooltip showing the date on the habit grid
- File organization under `src/components/goal-card/{count,checklist,habit}.tsx` + `src/components/goal-card/index.tsx` variant picker (follows `ARCHITECTURE.md` recommendation but exact shape is planner's call)

</decisions>

<specifics>
## Specific Ideas

- **"Bars moving is the product."** Every decision in this phase optimizes for that moment. Identical bar component across all three types, spring physics on width, `useOptimistic` on every mutation — all in service of the bar feeling alive the instant the user taps.
- **No punishing streak counter, ever.** The habit grid shows gaps; there is never a red X, never a "streak: 0", never a "you broke your streak" toast. `FEATURES.md §Anti-Features` and `PITFALLS.md §1` are unambiguous.
- **Early-month guard is research-locked, not optional.** Day-3 bars at 7% look worse than empty and demotivate users. The pace chip hides and the expected-line dims before day 5. `PITFALLS.md §4`.
- **Pace semantics are a small pure function with outsized emotional impact.** `expected = days_elapsed / days_in_month`; `paceDelta = round((percent - expected) × target)`. A single unit test fixture locks this forever.
- **Optimistic UI is mandatory, not a polish item.** Server round-trip must never block the bar animation. React 19 `useOptimistic` + Motion spring-physics width is the pattern.
- **The progress-bar replacement for Radix Progress is a known research flag.** `STATE.md §Blockers/Concerns`: "Motion + shadcn Progress customization needs a pattern pass to avoid CLS regression." Planner/researcher must address this before implementation — do not treat it as polish.
- **Dashboard query budget: < 30 lines of SQL.** `PITFALLS.md §3` acceptance test. If the dashboard query grows beyond that, the schema is wrong, not the query.
- **"Log for earlier day" is a real feature, not a nice-to-have.** PROG-04 is scoped to Phase 2. Both habit grid cells and count card kebab menus must surface it.
- **Type is immutable after creation.** Editing type would require re-validating all historical progress against a new shape. Forbid it at the UI layer; enforce at the service layer.

</specifics>

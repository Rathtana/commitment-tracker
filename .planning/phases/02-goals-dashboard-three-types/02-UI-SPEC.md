## Brand Tokens — Extension to `src/app/globals.css`

Phase 1 tokens stay unchanged. Phase 2 ADDS two new color families (pace chip + habit grid states). It does NOT add any `--spacing-*` or `--radius-*` overrides (02-RESEARCH.md Pitfall 10 / STATE.md Plan 01-05 lesson — these collide with Tailwind v4 core utilities).

Append inside the existing `@theme` block in `src/app/globals.css`:

```css
/* --- Phase 2 additions: pace chip + habit grid states --- */

/* Muted amber for "behind by N" — intentionally NOT red (PITFALLS §1: no shame, no punishment) */
--color-warning:             oklch(70% 0.130 75);     /* amber-500 */
--color-warning-foreground:  oklch(26% 0.055 50);     /* amber-950 — dark text on amber for WCAG AA */
--color-warning-muted:       oklch(96% 0.030 88);     /* amber-50 — chip background */

/* Muted emerald echo for "ahead by N" — distinct from --color-primary so the chip doesn't
   read as a duplicate CTA. Reuses the emerald hue at a lighter / more muted chroma. */
--color-success:             oklch(66% 0.120 162);    /* emerald-500 desaturated */
--color-success-foreground:  oklch(26% 0.060 161);    /* emerald-950 */
--color-success-muted:       oklch(95% 0.025 164);    /* emerald-50 — chip background */

/* Habit grid — reuse existing tokens:
   - Hit cell fill        → var(--color-primary) (emerald-600)
   - Miss cell fill       → var(--color-muted)   (zinc-100)
   - Today ring outline   → var(--color-ring)    (emerald-400)
   - Future dimmed        → var(--color-muted) at opacity 0.5 */
```

Matching dark-mode overrides inside the existing `@media (prefers-color-scheme: dark) { @theme { ... } }` block:

```css
--color-warning:             oklch(75% 0.140 78);     /* amber-400 — lighter in dark */
--color-warning-foreground:  oklch(20% 0.040 50);
--color-warning-muted:       oklch(22% 0.045 75);     /* amber-900 tint */

--color-success:             oklch(70% 0.125 161);
--color-success-foreground:  oklch(15% 0.050 161);
--color-success-muted:       oklch(22% 0.050 162);    /* emerald-900 tint */
```

**Rules the executor MUST respect when editing `globals.css` (STATE.md Plan 01-05 lesson):**

1. Do NOT define `--spacing-xs`, `--spacing-sm`, `--spacing-md`, `--spacing-lg`, `--spacing-xl`, `--spacing-2xl`, `--spacing-3xl` — they shadow Tailwind v4 core and break `max-w-*`, `w-*`, `p-*`, `gap-*`.
2. Do NOT rename existing `--color-*` tokens. Add new ones only.
3. Do NOT add `--radius-*` tokens; Tailwind v4 ships `rounded-sm`/`md`/`lg`/`full` already.
4. For one-off spacing (grid gap, pace-chip padding), use arbitrary values (`gap-[0.25rem]`) — not a token.

## Spacing Scale

Inherited from Phase 1 unchanged (uses Tailwind v4 default `--spacing-*` utility set; no custom tokens). All multiples of 4 from the default utility scale.

| Token | Value | Phase 2 Usage |
|-------|-------|---------------|
| `p-1` / `gap-1` | 4px | Habit grid cell inner spacing, icon-to-text gap inside kebab items |
| `p-2` / `gap-2` | 8px | Habit grid row gap, task-row inner gap, pace-chip `px-2 py-0.5` |
| `p-4` / `gap-4` | 16px | Goal card inner padding (horizontal), field-to-field gap in Dialog, dashboard vertical rhythm between cards |
| `p-6` / `gap-6` | 24px | Goal card vertical padding, Dialog content padding, empty-state section breaks |
| `p-8` / `gap-8` | 32px | Dashboard top padding on desktop, empty-state "Add your first goal" spacing |
| `p-12` | 48px | Empty-state headline top offset on desktop |
| `p-16` | 64px | (unused in Phase 2 — reserved for Phase 3 month-nav chrome) |

Exceptions:

- **Touch targets: ≥ 44px** on every interactive element (Button, kebab trigger, Checkbox row, habit-grid cell, stepper `-`/`+` buttons). Apply `h-11` or `min-h-11 min-w-11` where the element's visual size is smaller — POLSH-01 compliance laid early.
- **Habit grid cells:** 36px × 36px visual (`h-9 w-9`), wrapped in a `min-h-11 min-w-11` button padding container so the clickable area meets 44px even though the visible cell is 36px. This is the only place the visual/touch size diverges.
- **Progress bar height:** `h-2` (8px) — narrow enough that it doesn't dominate the card, wide enough to show `scaleX` animation clearly. `rounded-full`.
- **Pace chip:** `px-2 py-0.5` (8px horizontal / 2px vertical) — compact inline text label.

Inherits Phase 1 palette. Phase 2 adds two new families (warning/amber + success/emerald-muted) for the pace chip and reuses existing tokens for the habit grid.

| Role | Light Value | Dark Value | Phase 2 Usage |

| Warning (pace: behind) | `--color-warning` (amber-500) + `--color-warning-muted` chip bg | amber-400 | Pace chip text + background when `pace==='behind'` — NEVER used for bar fill, NEVER used for validation errors |
| Success (pace: ahead) | `--color-success` (emerald-500 muted) + `--color-success-muted` chip bg | emerald-400 muted | Pace chip text + background when `pace==='ahead'` |

### Accent Reserved-For List (EXPLICIT — downstream auditor checks this)

`--color-primary` (emerald-600) is used ONLY for:

1. Progress-bar fill (`<motion.div>` `bg-primary`) — the hero surface for the product's core value
2. Primary `+1` button on count cards (background fill)
3. Submit buttons inside create/edit goal Dialog (shadcn Button default variant)
4. "Add your first goal" empty-state CTA
5. Habit grid "hit" cells (cell background)
6. Focus-visible ring on all interactive elements (via `--color-ring` emerald-400 — lighter sibling)

`--color-primary` is NOT used for:

- Decorative borders, card backgrounds, goal titles, body copy
- Checkbox fill when checked (uses shadcn default — emerald check inside border, no primary flood)
- Pace chip background (uses amber/success muted — pace is NOT a primary action)
- Today's ring on habit grid (uses `--color-ring` (emerald-400) at 2px outline — distinct from the fill)
- Any hover state (hover uses `--color-primary-hover` only on interactive-filled elements; other hovers are `bg-muted` or no background change)

### Destructive Reserved-For List

`--color-destructive` (red-600) is used ONLY for:

1. Delete goal `AlertDialog` confirm button
2. Field validation errors inside the create/edit goal Dialog (border + error text)
3. Optimistic-rollback inline Alert (baseline behavior; Phase 4 hardens the UX)

`--color-destructive` is NEVER used for:

- Pace "behind" state (this is the core product-psychology rule — PITFALLS §1)
- Habit "miss" cell (miss = empty muted, not red — PITFALLS §1)
- Empty states (empty = neutral prompt, not error)
- Goal deletion itself — the DELETE is confirmed via AlertDialog; the card doesn't turn red first

### 60 / 30 / 10 Split Verification

- **60% background:** Dashboard page `bg-background` (zinc-50 / zinc-950). The viewport is overwhelmingly this neutral surface.
- **30% surface:** Goal cards (`bg-card`), progress-bar rail (`bg-muted`), Dialog content (`bg-card`), habit miss-cells (`bg-muted`). Together these occupy ~30% of the painted surface above the fold.
- **10% accent:** Progress-bar fills, `+1` buttons, habit hit-cells, CTAs. The parts that DRAW THE EYE.

## Interaction Contract

### Progress Bar (shared across all three card types — the product's hero surface)

**Component:** `src/components/progress-bar.tsx`
**Source:** 02-RESEARCH.md Pattern 3 (mandatory — this is the STATE.md research flag)

| Behavior | Spec |
|----------|------|
| Dimensions | `h-2 w-full rounded-full bg-muted` (rail track) |
| Fill | `absolute inset-y-0 left-0 w-full origin-left rounded-full bg-primary` on a `<motion.div>`; `style={{ willChange: 'transform' }}` |
| Animation property | `scaleX` (NOT `width`) — CLS-safe because `transform` lives on the compositor |
| Animation curve | Spring: `{ type: 'spring', stiffness: 140, damping: 22, mass: 0.6 }` |
| Initial render | `initial={false}` — never re-animate from 0 on mount (prevents flicker on re-render) |
| Reduced motion | `useReducedMotion()` → `{ duration: 0 }` when `true` — instant update, no animation |
| Expected-line tick overlay | Absolute `w-px h-full bg-foreground/40` positioned at `left: ${expected * 100}%` — rendered only when `expected` prop supplied AND `pace !== 'warming-up'` |
| ARIA | `role="progressbar"` + `aria-valuemin=0` + `aria-valuemax=100` + `aria-valuenow=Math.round(percent*100)` + `aria-label="{goal title}: {done} of {total} ({percent}%)"` |
| Color shift by pace | NEVER — bar stays emerald regardless of pace (D-25, PITFALLS §1). Behind/ahead communicated via chip + tick ONLY |
| Fill when `percent === 0` | `scaleX: 0` — rail visible, no fill |
| Fill when `percent === 1` | `scaleX: 1` — full rail, no overflow animation |
| Fill when `percent > 1` (edge case — e.g., over-target count) | Clamped to `Math.min(1, percent)` at the component boundary |

### Undo Toast Interaction

| Behavior | Spec |
|----------|------|
| Trigger | Every successful progress mutation (see "Undo Toast" copy table above) |
| Duration | 6000ms (D-32) |
| Action button | "Undo" — reverses the specific mutation via its server-generated `undoId` |

| Dismiss behavior | After 6s auto-dismiss OR on click outside OR on "Undo" click |

### Card Entry Animation (nice-to-have, not load-bearing)

| Event | Behavior |
|-------|----------|
| Dashboard first render | No entrance animation — cards appear instantly (server-rendered, not dynamic) |
| New goal created via Dialog | `AnimatePresence` + Motion's `layout` prop: new card slides in from top with 200ms ease-out, existing cards shift down |
| Goal deleted | Card fades out 150ms + remaining cards shift up via `layout` animation |
| Optimistic progress update | Bar `scaleX` springs; no card-level motion |
| Reduced-motion | All `layout` animations bypass (`duration: 0` via `useReducedMotion`) |

### Keyboard Navigation

- **Tab** traverses: kebab → progress bar (skipped; progressbar role not tabbable) → type-specific interactive elements (`+1` → stepper; or task rows; or habit cells) → next card.
- **Enter** / **Space** on focused element = activation.
- **Arrow keys inside habit grid:** navigate between cells (Roving tabindex pattern). Up/Down/Left/Right move focus; Enter/Space toggle the focused cell.
- **Escape** closes any open Dialog / AlertDialog / Popover / DropdownMenu. Does NOT dismiss the undo toast (Sonner uses its own ESC handling).
- **Undo toast keyboard:** Focus does NOT automatically move to the toast (would break flow). Power users can `Ctrl+Z` as a nice-to-have — NOT in Phase 2 scope; note for Phase 4.

### Focus-Visible Ring

All focusable elements inherit Phase 1's `focus-visible:ring-2 focus-visible:ring-ring focus-visible:ring-offset-2` (emerald-400, 2px, 2px offset, `ring-offset-background`). No overrides needed — shadcn primitives wire this automatically.

Habit grid cells must have visible focus ring AS WELL AS the "today" ring when focused on today — the two rings compose (focus ring 2px + today ring 2px); use `ring-offset-card` on the today ring so the focus ring sits outside it.

## Accessibility

| Requirement | Implementation |
|-------------|----------------|
| Minimum touch target | 44px × 44px on every interactive element (Button `h-11`, kebab icon button 44x44, stepper buttons 44x44, habit grid cells wrapped in 44x44 padding container) |
| Focus indicator | emerald-400 2px ring with 2px offset (inherited from Phase 1) |
| Progress bar ARIA | `role="progressbar"` + `aria-valuemin`/`aria-valuemax`/`aria-valuenow` + descriptive `aria-label` (see spec above) |
| Progress bar live region | NO — constant value updates would spam screen readers. Updates are tied to user action which has its own feedback (toast) |
| Habit grid cell ARIA | Every cell has explicit `aria-label` including date + state ("April 3 — done" / "April 3 — not done" / "April 3 — today, not yet done") — D-40 |
| Roving tabindex on habit grid | Arrow-key navigation between cells; only the focused cell has `tabIndex=0`, others have `tabIndex=-1` |
| Dialog focus trap | shadcn Dialog's Radix primitive handles focus trap + focus restore on close |
| AlertDialog role | `role="alertdialog"` — Radix handles |
| Toast live region | Sonner uses `role="status"` + `aria-live="polite"` — announces "Logged +1" etc. naturally |
| Toast Undo button | Tab-reachable within toast; `aria-label="Undo last action"` on the action button |
| Color contrast | All combinations verified ≥ WCAG AA: emerald-600 on white (AA), zinc-500 on zinc-50 (AA — for muted captions), amber-950 on amber-50 (AAA — pace chip), white on emerald-600 (AA+) |
| Reduced motion | `useReducedMotion` hook on progress bar + card layout animations → instant updates with no spring |
| Screen-reader order | Card DOM order: heading → progress bar → type-specific content → raw-count caption. Matches visual order. |
| Semantic HTML | `<main>` on dashboard page, `<section>` per goal card optional, `<h1>` for page title, `<h3>` for goal titles inside cards |
| Keyboard-only flow validation | User can: open "New goal" → pick type → fill fields → submit → focus returns to trigger → Tab to new card → kebab → Edit → edit → save. All without mouse. |

## Layout

### Dashboard Page (`/dashboard` — Phase 2 replaces the Phase 1 landing stub at `/`)

Primary focal point: the FIRST goal card's progress bar. The user's eye should land there. The page header ("April 2026" + "New goal" button) sits above the fold as secondary visual weight.

```
[viewport — bg-background]
  │
  └─ <main> — max-w-[720px] mx-auto px-4 py-8 (mobile) / py-12 (desktop) flex flex-col gap-4
       │
       ├─ <header> — flex items-center justify-between mb-2
       │   ├─ <h1>{Month Year}</h1>                              ← 24px semibold
       │   └─ <div>                                              ← right cluster
       │       ├─ <Button variant="default" size="default">     ← "New goal" primary
       │       │    <Plus size={16} /> New goal
       │       │  </Button>
       │       └─ <Button variant="outline" size="sm">Log out</Button>  ← logout inherited Phase 1
       │
       ├─ [if goals.length === 0]
       │   └─ <EmptyState />                                     ← see Empty State Layout below
       │
       └─ [if goals.length > 0]
           └─ <section className="flex flex-col gap-4">          ← gap between cards
                ├─ <GoalCard goal={...} />                       ← repeated
                ├─ <GoalCard goal={...} />
                └─ …
```

- Container: `max-w-[720px]` centered. On mobile, `px-4` means full-width minus 16px margin each side. On desktop, extra whitespace flanks the column.
- Cards stack vertically with `gap-4` (16px between cards). NO dividers, no tabs, no drill-down (DASH-01).
- Page header stays static (not sticky in Phase 2 — reserve sticky for Phase 3's month nav).
- If the user has many goals, the page scrolls naturally — no "load more", no pagination.

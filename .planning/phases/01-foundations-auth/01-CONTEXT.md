# Phase 1: Foundations & Auth - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Deploy a Next.js 16 + Supabase + Drizzle scaffold with complete email/password authentication (signup → email verification → login → logout → password reset) and a schema-locked foundation for the three-goal-shape data model. Phase 1 delivers:

- A working auth loop that a new user can complete in under 2 minutes
- The `public.users` mirror table (with `timezone` column) and the `goals` polymorphic parent table (with month-pinned `DATE` CHECK constraint) — both under RLS
- A locked timezone strategy validated by a DST/month-boundary Vitest test suite against pure time functions

**Not in scope:** user-facing goal creation UI, progress logging UI, dashboard goal rendering, child tables (`tasks`, `habit_check_ins`), month navigation. Those are Phase 2 and Phase 3.

</domain>

<decisions>
## Implementation Decisions

### Phase Scope
- **D-01:** Phase 1 is strict ROADMAP scope — auth + schema + timezone test suite. No user-facing goal UI. Research SUMMARY's "count-goal walking skeleton" is rejected for Phase 1; Phase 2 owns the whole create-goal-and-log loop end-to-end.
- **D-02:** Post-login landing page is a minimal stub: `"Welcome, {email}"` + logout button + `"Goals coming in Phase 2"` placeholder text. No `/dashboard` route shell, no dynamic month routing, no empty-state card component. Intentionally bare.

### Schema & Migrations
- **D-03:** Phase 1 migration ships exactly two tables: `public.users` (mirror of `auth.users` holding per-user profile fields including `timezone`) and `goals` (the polymorphic parent).
- **D-04:** `tasks` and `habit_check_ins` child tables are deferred to Phase 2 where they are actually used. Phase 1 avoids dead schema.
- **D-05:** `goals.month` is `DATE NOT NULL` with `CHECK (EXTRACT(DAY FROM month) = 1)` — pinned to the first-of-month at the DB layer so the invariant is impossible to violate from any app code path.
- **D-06:** `public.users`: `id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE`, `timezone TEXT NOT NULL DEFAULT 'UTC'`, plus any standard `created_at`/`updated_at` timestamps.
- **D-07:** `goals.user_id UUID NOT NULL REFERENCES public.users(id) ON DELETE CASCADE`. Real FK (not just an unconstrained UUID column). Referential integrity at the DB level.
- **D-08:** A trigger on `auth.users` INSERT auto-populates a corresponding `public.users` row (standard Supabase mirror pattern). Users cannot INSERT into `public.users` directly.
- **D-09:** Migrations are generated by `drizzle-kit generate` (from Drizzle schema files) and applied via Supabase CLI (`supabase db push` / `supabase migration up`) for local/staging/prod propagation. Both tools participate: Drizzle owns generation, Supabase CLI owns environment propagation.

### Timezone Strategy
- **D-10:** Server-side timezone math uses `date-fns-tz` (aligns with `date-fns` already in the stack). No Temporal API for v1 — polyfill cost and immaturity are not worth it; revisit ~2027 when polyfill becomes droppable.
- **D-11:** Each user's IANA timezone lives on `public.users.timezone`. This is the authoritative source. Server reads it per request. No per-request `tz` passed in headers/cookies.
- **D-12:** Timezone is captured at signup: the signup form reads `Intl.DateTimeFormat().resolvedOptions().timeZone` client-side and passes it to the signup server action. Defaults to `'UTC'` if detection fails or returns falsy.
- **D-13:** A pure isomorphic function `today(now: Date, userTz: string): string` returns a `'YYYY-MM-DD'` local date. Imported by server code (authoritative) and later Phase 2 client code (optimistic UI). Single source of truth for "what day is it for this user."
- **D-14:** A pure isomorphic function `monthBucket(now: Date, userTz: string): Date` returns the first-of-month `DATE` for the user's local day. Used to compute `goals.month` on any write and to query "current month" on any read.

### Authentication Flows
- **D-15:** Email/password only in Phase 1. OAuth (Google/Apple) is explicitly deferred per REQUIREMENTS.md OAUTH-01 (v2+).
- **D-16:** Email verification is **required before login**. Unverified users attempting to log in see a "please verify your email" message and cannot proceed. Matches success criterion #1.
- **D-17:** Password reset links: 15-minute expiry (per success criterion #3), single-use enforced by Supabase on code exchange. No bespoke IP rate limiting in Phase 1 — ROADMAP Phase 4 POLSH work adds it.
- **D-18:** Auth cookies configured via `@supabase/ssr` middleware template defaults: `HttpOnly` + `Secure` + `SameSite=Lax`, token refresh on each server render. `createServerClient` in RSC/server actions, `createBrowserClient` in client components. No custom `Domain` or `Path` override.
- **D-19:** Email delivery uses Supabase's built-in SMTP for v1. Sender: `noreply@mail.app.supabase.io`. Rate-limited to 4 emails/hour on free tier — acceptable for launch and builder's daily use. Swap to Resend or Postmark before any real public marketing push.

### Row-Level Security
- **D-20:** Policies are written using Drizzle's `crudPolicy()` helpers co-located with the schema definitions. Drizzle is the single source of truth for both tables and policies; `drizzle-kit` generates the policy SQL into migrations.
- **D-21:** `public.users` RLS: `SELECT` and `UPDATE` restricted to `id = auth.uid()`. No user-driven `INSERT` (trigger handles it). No `DELETE` (cascade from `auth.users` handles it).
- **D-22:** `goals` RLS: all four CRUD operations restricted to `user_id = auth.uid()`. Past-month write protection is a Phase 3 concern — not enforced at the RLS layer in Phase 1.

### Testing Surface
- **D-23:** Vitest test suite with fake timers covers the timezone strategy. Fixtures must include at minimum: 11:30 PM on the last day of a month at UTC-8, UTC+13, and UTC+0; DST spring-forward day; a leap-year Feb 28/29 boundary; New Year's Eve midnight. Locks the pitfall from research/PITFALLS.md §1.
- **D-24:** Tests target the pure `today()` and `monthBucket()` functions directly — no database, no middleware, no Next.js runtime. Fast and deterministic; runs in CI on every commit.

### Claude's Discretion
- Exact shape of `middleware.ts` (follow `@supabase/ssr` official template)
- Copy, layout, and visual polish of login/signup/reset/verify pages beyond "functional and clean with shadcn primitives"
- Whether auth forms use shadcn `<Form>` wrapper or plain `<form>` (must use react-hook-form + zod per CLAUDE.md stack)
- Drizzle schema file organization (one file vs split per table)
- ESLint/Prettier specifics beyond `eslint-config-next` + `prettier-plugin-tailwindcss`
- Vitest config shape (jsdom vs node environment for which tests)
- Directory layout under `src/` (conventional to be decided during planning)

</decisions>

<specifics>
## Specific Ideas

- **Stack is locked in CLAUDE.md** — do not deviate during planning or execution. The versions pinned there are verified via live `npm view` on 2026-04-17.
- **Landing page is intentionally bare.** "Welcome, {email}" + logout + "Goals coming in Phase 2" placeholder is the whole shape. Resist the temptation to ship a dashboard shell — that's Phase 2's job.
- **Timezone IS the foundation.** Pure function pattern, tested with fixtures, stored per user. Research flagged this as "expensive to retrofit" — treat the test suite as a first-class deliverable, not a bonus.
- **The schema must be impossible to violate from above.** `goals.month` CHECK constraint (day = 1) means no app code path can ever write a non-first-of-month value. That invariant is the point of Phase 1.
- **RLS is defense in depth, not the only line.** Services also check ownership — RLS catches bugs in service code; services catch bugs in RLS.

</specifics>
